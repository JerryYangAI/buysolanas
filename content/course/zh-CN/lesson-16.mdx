## Solana 程序（智能合约）概念入门

在以太坊上，它们叫智能合约。在 Solana 上，它们叫**程序**。虽然概念类似——在区块链上运行的代码——但 Solana 的架构有本质的不同。即使你从不写代码，理解这种架构也很有价值，因为它帮助你评估项目和理解应用的底层运作方式。

### 账户模型

Solana 最大的架构差异是其账户模型。在以太坊上，智能合约将代码和数据存储在一起。在 Solana 上，这两者是分离的：

- **程序** —— 只包含可执行代码，是无状态的。
- **账户** —— 存储所有数据。程序从账户中读取数据并向账户中写入数据。

可以把它想象成餐厅：程序是菜谱（指令），账户是食材和盘子（数据）。菜谱本身不存储任何食物——它只描述如何处理食物。

**为什么这很重要：**
- 程序可以在许多不同的数据账户间复用。
- 多个程序可以读取同一个账户的数据。
- 这种分离使 Solana 能够并行处理交易，这是其速度的关键。

### 核心概念

**程序 ID：** Solana 上的每个程序都有一个唯一地址（程序 ID）。当你与 Jupiter 交互时，你的钱包向 Jupiter 的程序 ID 发送包含特定指令的交易。

**指令：** Solana 上的交易包含一个或多个指令。每个指令指定：
- 调用哪个程序。
- 程序应该读取或修改哪些账户。
- 传递什么数据（参数）。

**所有权：** Solana 上的每个账户都由一个程序拥有。只有拥有该账户的程序才能修改其数据。系统程序拥有普通 SOL 账户，代币程序拥有代币账户，依此类推。

**租金：** Solana 上的账户必须维持最低 SOL 余额以覆盖存储成本。这叫做"免租金豁免"。大多数应用会预付此金额，使账户可以永久存在。

### 常用 Solana 程序

| 程序 | 用途 |
|------|------|
| 系统程序 | 创建账户、转账 SOL |
| 代币程序 | 创建和管理 SPL 代币 |
| 关联代币账户程序 | 创建与钱包关联的代币账户 |
| 备忘程序 | 在交易中附加消息 |

这些内置程序处理所有 Solana 应用构建的基础操作。

### Anchor 框架

Anchor 是构建 Solana 程序最流行的框架。它通过以下方式简化开发：

- **减少样板代码** —— Anchor 自动处理账户的序列化、反序列化和验证。
- **提供宏** —— Rust 宏生成常见代码模式，使程序更短、更易读。
- **标准化模式** —— Anchor 建立了使程序更易审计和理解的惯例。

Solana 上大多数 DeFi 协议、NFT 平台和 dApp 都使用 Anchor 构建。当你在审计报告或文档中看到"Anchor 程序"时，表示该项目使用了这个框架。

### 交易的执行流程

1. 你在 Jupiter 网站上点击"Swap"。
2. 你的浏览器构建一个包含 Jupiter 程序指令的交易。
3. 交易指定使用哪些账户（你的代币账户、流动性池等）。
4. 你的钱包用私钥签署交易。
5. 交易被发送给验证者，验证者执行程序代码。
6. 程序读取并修改指定的账户。
7. 验证者在下一个区块中确认交易。

### 为什么这些知识很重要

即使作为非开发者，理解账户模型也能帮助你：

- **阅读交易详情** —— 你可以在 SolScan 上理解交易实际做了什么。
- **评估安全性** —— 了解程序和账户如何交互有助于评估项目风险。
- **理解错误** —— 常见错误如"insufficient funds for rent"会更有意义。
- **评判创新** —— 你可以更好地判断项目的技术声明是否现实。
